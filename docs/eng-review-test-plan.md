## Key Interactions to Verify

### Reading flow
- Open chapter on desktop, drop cap renders, marginalia visible to the right of body
- Open chapter on mobile, footnotes appear as expandable cards, no horizontal scroll
- "See also" links at chapter end navigate to related chapter
- Continue reading link at chapter end advances to next chapter
- View transition between chapters feels like a single document

### Notebook flow
- Highlight a passage by selecting text, click "Save highlight", confirm appears in drawer
- Add a note to an existing highlight, edit it, save, confirm persistence on reload
- Add a "doubt" (open question) tied to a passage
- Delete a highlight, confirm it's gone after reload
- Open `/notebook/` global view, see all entries grouped by work

### Export flow
- "Open in Claude" button: notebook is short → opens claude.ai/new with prompt prefilled
- "Open in Claude" button: notebook is long → button transforms to "Download as Markdown" with one-line instruction
- "Open in ChatGPT" button: same length-check pattern
- "Download Markdown" button: downloads `notebook.md` with proper formatting
- "Download JSON" button: downloads `notebook.json` with structured data

### MCP integration (run via Claude Desktop or stdio harness)
- `npx @falsafa/mcp` boots without errors
- `list_works()` returns 37 entries with full metadata
- `read_chapter(slug, 1)` returns chapter content as markdown
- `search_corpus("courage")` returns matches from English-translated chapters only
- `search_corpus("courage", { scope: "all" })` returns matches across original + translit + translation
- `find_related(slug, n)` returns 3-5 related chapters
- `compare_works(slug_a, slug_b)` returns side-by-side chapters

### Byline rendering
- Work with `content_type: translation` shows "Translated by Thothica" on chapter pages
- Work with `content_type: transliteration` shows "Transliterated by Thothica"
- Work with `content_type: original` shows "Curated by Thothica"
- All bylines link to thothica.com

## Edge Cases

- Notebook with 0 entries — empty state copy and CTA
- Notebook with 200 entries — pagination or virtual scroll, no jank
- Chapter with empty content — fallback message, no broken layout
- Chapter with 50,000 characters (largest works) — paginated or chunked render
- Chapter title containing diacritics or RTL characters (Urdu) — proper rendering and slug
- Chapter image fails to load (404 or 500) — graceful fallback to text-only render
- User has IndexedDB disabled (privacy mode) — notebook UI shows "Storage unavailable" with explanation
- User exhausts IndexedDB quota — clear error + export-and-clear flow
- Two browser tabs editing notebook simultaneously — last write wins, no corruption
- User navigates away mid-export — no orphaned download
- MCP `read_chapter("nonexistent-work-slug", 1)` — typed `MCPError { code: "WORK_NOT_FOUND" }`
- MCP `read_chapter(valid_slug, 999)` — typed `MCPError { code: "CHAPTER_OUT_OF_RANGE" }`
- MCP `search_corpus("")` — returns empty array, not error
- Build pipeline run twice — second run is byte-identical to first (idempotency)
- Build pipeline with new work added — only new work generates images, existing assets untouched

## Critical Paths

These flows must work end-to-end before launch.

1. **First read.** Land on home → click featured work → click chapter 1 → read to end → click see-also → land on related chapter. (No reload, no JS errors, view transition smooth.)

2. **Notebook round-trip.** Read a chapter → highlight three passages → add a doubt → reload page → confirm all persisted → click "Open in Claude" → confirm claude.ai opens with the notebook prefilled.

3. **MCP round-trip.** User installs `@falsafa/mcp` in Claude Desktop → asks Claude "what works are by Cynewulf?" → Claude calls `list_works({ author: "cynewulf" })` → returns Andreas, Elene, Juliana → asks Claude "show me chapter 1 of Andreas" → Claude calls `read_chapter("cynewulf-andreas-07b573", 1)` → returns content → Claude displays.

4. **Build pipeline.** Fresh clone → `pnpm install` → `pnpm convert` → `pnpm cross-link` → `pnpm images --skip-existing` → `pnpm build` → site builds in under 2 minutes, all 1,673 chapter pages exist, MCP package builds, no warnings.

## MCP Black-Box Eval Suite

Spawn a fresh LLM context (default Claude Sonnet via OpenRouter, configurable to
Hermes / GPT / etc.) with ONLY the falsafa MCP server attached. No corpus context.
Pose corpus-related questions. Score whether the LLM successfully answers using
the MCP tools alone. If a frontier LLM with no priors can't navigate the corpus
to answer a real question, the tools are wrong, not the LLM.

**~16 eval cases** covering:

1. Factual lookup — "What works did Cynewulf write?"
2. Cross-text discovery — "Where in the corpus is the concept of dharma discussed?"
3. Author comparison — "Compare how Cynewulf and Iqbal treat the divine"
4. Citation precision — "Quote me a passage about courage from Andreas with line numbers"
5. Onboarding discovery — "What's a good first work for someone new to Old English literature?"
6. Negative case — "Find me works in Tamil" (should return empty cleanly, no hallucination)
7. Cross-language reasoning — "Find passages from Iqbal that resonate with the Andreas opening"
8. Difficulty-aware retrieval — "Show me the most accessible Sanskrit work in the corpus"
9. Content-type-aware retrieval — "Show me works in their original language only"
10. Translation-aware retrieval — "Show me only Thothica's translations"
11. Era-bounded query — "What philosophical works exist from the medieval period?"
12. Genre-bounded query — "List all literary works"
13. Multi-step reasoning — "Of all works that mention 'divine,' which has the highest word count?"
14. Edge: out-of-range — "Show me chapter 999 of Andreas" (graceful error handling)
15. Edge: invalid filter — "List works in the language called 'martian'" (graceful empty)
16. Citation fidelity — "Give me the exact opening line of Andreas" (must be byte-accurate)

**Judging:** rubric of (a) correct tool calls, (b) factually correct answer, (c)
citation-backed. Judge model = Claude Opus or GPT-4-class. Failure threshold:
any eval scoring below 2/3 fails CI.

**CI:** GitHub Actions on every PR to `apps/mcp/`, ~$2-5 per run, gated by
`EVALS_REQUIRED=true` repo secret so contributors without OpenRouter keys can PR.

**Failure mode:** 3+ evals failing across consecutive PRs signals tool-surface
redesign needed, not just bug fixes.
