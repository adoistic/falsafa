## Context

The user owns a corpus of 38 classical / humanities / philosophical works (1,673
chapters, ~4.7M tokens, mixed languages: Old English, French, German, Sanskrit,
Urdu, etc., with English translations) currently stored as a single ~21MB JSON
file exported from Neon Postgres. They want to:

1. Build a **premium reading experience** — mobile-first AND laptop-first, equally,
   designed for people to come and devour the texts.
2. Convert the JSON corpus into structured Markdown files (one source of truth on
   disk, no database).
3. Layer an **agentic RAG** on top of these Markdown files in the spirit of
   Karpathy's "no vector DB, just give the LLM tools to navigate" gist
   (https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).
4. Use **Nous Hermes via OpenRouter** as the LLM.
5. Treat this as a **side project** — Builder mode, not Startup mode.

## Phase 2B: Builder-Mode Questions (in progress)

### Q1: Who's it for, and what's the moment they don't forget?

**Answer:** Between a public site and an MVP. Free for anyone to read (no
auth, no payment). Open source. The user explicitly wants this to be a
**great open-source project**. Two surfaces:

1. **A public reading site** — anyone can land on it and devour the texts.
2. **An open-source MCP engine** — any LLM (Claude, ChatGPT, Hermes, etc.)
   can plug into the same corpus as a knowledge resource via MCP tools.

### Scope cut (user decision, mid-session):

- **No gated AI on the website for MVP.** No login, no approval flow, no AI
  inference handled by the site backend.
- **The website is purely a reader.** Static HTML/CSS/JS, no server-side
  state, no DB, no auth.
- **All AI behavior moves to MCP.** Users who want AI reasoning over the
  corpus install the MCP server in their own LLM client (Claude Desktop,
  Cursor, Codex, etc.) and let *their* LLM do the work with their own keys.
- **Hosting is trivial.** Netlify or Vercel free tier handles the entire
  reading site. The MCP server is distributed as an npm package or via
  GitHub clone.

### Q2: What does the MCP server hand the LLM?

**Answer:** Research-agent-flavored. Beyond Karpathy's minimal ls/grep/read,
the MCP exposes higher-level tools that do real lifting: `summarize_chapter`,
`find_related`, `compare_works`, in addition to `list_works`, `list_chapters`,
`search_corpus`, `read_chapter`, `get_passage`, `get_metadata`. The MCP itself
is mildly agentic — it calls a configurable LLM (default Hermes via OpenRouter,
user-provided key) for the synthesis tools. The server runs locally on the
user's machine in their MCP client, so no central API costs.

### Q3: What does "premium reading" mean?

**Answer:** Editorial / literary review feeling — NYRB, LRB, Aeon, Marginalia
Review of Books. Strong serif typography hierarchy, drop caps on chapter
opens, generous line-height, footnotes as marginalia on desktop and
expandable cards on mobile. The page reads like a printed essay you respect.

### Q4: V0 shape — narrow-deep or broad-shallow?

**Answer (overridden by user mid-question):** All 38 works, fully detailed,
fully polished. Not narrow-flagship-only. The user explicitly redirected:
"do it on 38 works, you can sort of go through each of the works, absorb
them, and then we can plan out how we will build the MCP and website."

The website should look "absolutely brilliant" — bold, not minimalist.

**New requirement surfaced:** AI-generated **artistic images for every
work and every chapter**, with deliberate context engineering — a per-work
style guide that gives the corpus a coherent visual language. ~38 cover
images + ~1,673 chapter images. This is a real piece of the design.

### Q5: Reader's notebook with portable AI export

**Answer:** The notebook is the killer feature. Readers can:

- Highlight passages, save quotes, take notes
- Add **doubts** — open questions about what they're reading
- Organize all of this into nodes (per-work or per-theme notebooks)
- **Export to formats that open directly in any major LLM platform:** Claude
  (Project starter), ChatGPT (deep-link prompt or canvas), Gemini, Sarvam,
  Mistral, DeepSeek, plus universal Markdown + JSON.

Architecturally this is brilliant: zero backend, zero auth, zero sync. The
notebook is the user's, lives in their browser (localStorage + IndexedDB
for capacity), and travels with them to whichever LLM they trust. No
AI-provider lock-in. The site never sees their notes.

This composes perfectly with the no-auth + MCP-first design.

### Q6: Graphic stories as a second content vertical

**Answer:** The user runs Studio Whence (whence.studio) — a commercial
graphic-stories company they founded. They're releasing some of the
graphic stories on Falsafa as **public goods**. Some of those stories
are adapted from works in this corpus; others are standalone.

Implication: the site has two content types — **scholarly works** and
**graphic stories**. They share the site's design language (Studio Whence
aesthetic can inform image style throughout) but live at parallel routes.
Cross-linked: a chapter that has a graphic adaptation surfaces a "see also"
to that adaptation; a graphic story credits the source work. Studio Whence
attribution everywhere. Only stories explicitly cleared as public goods
are included.

This also gives the corpus a distinctive visual identity — the AI-generated
chapter imagery can be guided by Studio Whence's house style, making the
whole site feel like a unified Studio Whence + Falsafa cultural project.

## Phase 3: Premises

These are the load-bearing assumptions of the design. Confirming them locks
the architecture; pushing back rewires it.

**P1. JSON to Markdown is a one-time, programmatic conversion.**
We write `scripts/convert.ts`, run it once, commit ~25MB of Markdown files
to the repo. The Markdown becomes the source of truth. The JSON is archived
as a historical artifact. This eliminates runtime parsing and aligns with
"no DB."

**P2. The MCP server is a librarian, not a second LLM.** *(refinement of Q2)*
You picked "research-agent-flavored," which I want to refine in one specific
way: the MCP tools should return *structure and text*, not *LLM-generated
synthesis*. `summarize_chapter` returns the chapter content plus computed
stats (word count, reading time, key entities found via grep).
`find_related` returns structural neighbors (same author, same era,
keyword overlaps). `compare_works` returns both works' relevant chapters
side-by-side. The host LLM (Claude, ChatGPT, Hermes via OpenRouter,
whatever the user plugs in) does all the reasoning. This keeps the MCP
server stateless, requires zero API keys to run, has no inference cost,
stays model-agnostic, and is closer to Karpathy's actual gist while still
giving you most of the leverage of "research-agent-flavored." If you
disagree, we go back to Hermes-inside-the-MCP.

**P3. All AI imagery is baked at build time.**
~38 cover images + ~1,673 chapter images, generated once via Flux or
SDXL through Replicate (~$5-15 one-shot cost), checked into the repo or
pinned to a CDN. Re-run only when the corpus changes. Zero runtime image
generation, zero ongoing cost. Style is governed by a per-era prompt
template informed by Studio Whence's house aesthetic.

**P4. The reader's notebook lives in localStorage / IndexedDB only.**
No backend, no accounts, no sync. Notes are exported to portable formats
when the user wants to bring them to an LLM.

**P5. Graphic stories are a parallel content vertical at /graphic-stories.**
Same design language, separate IA. Cross-linked at the chapter level via
"see also." Only Studio Whence stories explicitly cleared as public goods
are included.

**P5b. Translations are attributed to Thothica.** *(added late in session)*
All current translations in works.json are produced by Thothica, another
AI startup the user runs. Every translated work surfaces a "Translated by
Thothica" credit (linkable). The metadata schema must accommodate future
states: native-English works (no translator), works translated by third
parties (with their attribution). Thothica's translations are released
as public goods on Falsafa, mirroring the Studio Whence relationship.

**P6. All 38 works ship at once, fully polished, before launch.**
Boil the lake. No public V0 with stub works. The launch moment is when
every chapter is rendered with editorial typography, has its image, has
working marginalia, and the MCP server is functional against the full
corpus.

## Phase 4: Approaches Considered

### Approach A: Astro Editorial Monorepo (Recommended)

- **Reading site:** Astro 5 with content collections. Astro is purpose-built
  for content-heavy editorial sites: zero-JS by default, ships actual HTML,
  has first-class Markdown/MDX, excellent fonts integration, easy view
  transitions for that "feels like a single document" reading experience.
- **MCP server:** Bun + TypeScript using `@modelcontextprotocol/sdk`. Same
  monorepo, same Markdown directory as the source. Distributed as an npm
  package: `npm install -g @falsafa/mcp` or `npx @falsafa/mcp`.
- **Repo:** single monorepo, `pnpm` workspace.
  ```
  falsafa/
    corpus/                  ← markdown source of truth (~25MB)
      works/{slug}/...
      graphic-stories/{slug}/...
    apps/
      site/                  ← Astro
      mcp/                   ← Bun + TS, ships to npm
    scripts/
      convert.ts             ← JSON→Markdown one-shot
      generate-images.ts     ← Replicate/Flux pipeline
      cross-link.ts          ← computes "see also" at build time
    works.json               ← archived original
  ```
- **Hosting:** Vercel for the site (Astro adapter, free tier). MCP runs
  locally on user's machine via npx.
- **Pros:** Best DX. Best static-site performance. Smallest stack to learn
  if you don't already know Next. One repo to clone, fork, contribute to.
- **Cons:** Astro is less culturally dominant than Next.js. Some readers
  who fork the project may prefer Next.
- **Reuses:** existing markdown content in `chapter.content` is already
  most of the way there. Astro content collections eliminate routing code.
- Effort: ~3-4 weekends with CC.

### Approach B: Two-Repo, Distribution-First

- `falsafa-corpus` — Markdown only. Becomes the canonical artifact.
  Other people can fork or include it in their own projects. Git submodule
  into the site and MCP repos.
- `falsafa-site` — Next.js 15 App Router, fully static via `output: 'export'`.
- `falsafa-mcp` — TypeScript npm package.
- **Pros:** Cleanest separation. The corpus stands alone as a public good.
  Easier for others to consume just the corpus without the site or MCP.
- **Cons:** More setup overhead. Three repos to coordinate. Git submodules
  are notoriously friction-y. Slower to a shipping demo.
- Effort: ~5-6 weekends with CC.

### Approach C: Eleventy + Tiny Python MCP

- Reading site: Eleventy (the simplest possible SSG).
- MCP: ~150-line Python server using fastmcp.
- **Pros:** Lowest tech-stack footprint. Both pieces are tiny enough to
  read in one sitting.
- **Cons:** Hardest path to "absolutely brilliant" visuals — Eleventy's
  styling story is DIY, no opinionated component model, harder to keep
  the editorial design coherent across all 38 works. Two languages adds
  contributor friction.
- Effort: ~3 weekends but with significant front-end yak-shaving.

**RECOMMENDATION: Approach A (Astro Editorial Monorepo).** It minimizes
plumbing, maximizes typography craft, and keeps the corpus, site, and MCP
in one place where they can evolve together. The two-repo split (B) is
something we can do later if the corpus becomes important enough to live
on its own — premature now.

Frontmatter on each `chapters/*.md`:

### MCP toolset (librarian flavor — P2)

```
list_works(filter?: { era, author, genre, language, translator })
list_chapters(work_slug)
list_authors() / list_eras() / list_genres() / list_languages()
get_metadata(work_slug)         → returns frontmatter + manifest entry
read_chapter(work_slug, n)      → returns full markdown of one chapter
get_passage(work_slug, n, line_range)  → precise citations
search_corpus(query, scope?)    → ripgrep over corpus, returns matches
                                  with file paths, line numbers, context
find_related(work_slug, n)      → returns chapter slugs whose
                                  frontmatter `related` field overlaps
                                  (computed at build time, no LLM)
compare_works(slug_a, slug_b, topic?) → returns relevant chapters from
                                         each, side-by-side, no synthesis
list_graphic_stories(filter?)
read_graphic_story(slug)        → returns story metadata + panel URLs
```

The MCP server is a thin TypeScript wrapper around the filesystem and a
manifest. Zero LLM calls. Zero state. Zero API keys. Distributable as a
single npm package.

### AI imagery pipeline

- **Cover images (38):** Replicate Flux dev, painterly per-era style,
  ~1024x1024. Per-era prompt template lives in
  `scripts/image-prompts/{era}.json`, informed by Studio Whence's house
  aesthetic. Estimated cost: ~$0.50 total.
- **Chapter images (~1,673):** Same Flux dev, smaller resolution
  (768x512), prompt seeded with chapter title + first 200 chars of
  content. Estimated cost: ~$5-10 total.
- All outputs WebP, committed to the corpus directory under
  `assets/`. Re-runnable via `pnpm run images`.
- Total budget: ~$15 one-shot.

### Reader's notebook (P4)

- **Storage:** IndexedDB (localStorage too small for highlighted-passage
  corpus). Schema: `{ id, work_slug, chapter_n, anchor: { line_start,
  line_end }, kind: "highlight" | "note" | "doubt", body, tags,
  created_at }`.
- **UI:** Right-side drawer that slides in. Shows all notes for the
  current chapter, plus a global notebook view at `/notebook`.
- **Export formats** at `/notebook/export`:
  - `notebook.md` — universal Markdown (works in any LLM as a paste)
  - `notebook.claude.md` — Claude Project starter (with system prompt
    pre-filled to act as a study partner)
  - `notebook.chatgpt.url` — deep-link prompt URL
  - `notebook.gemini.md` — Gemini Gem starter
  - `notebook.json` — structured export for Sarvam, Mistral, DeepSeek,
    or any LLM that takes JSON context
  - `notebook.mcp.json` — context bundle the user can drop into the
    Falsafa MCP for cross-referencing while chatting
- **No backend.** No analytics on notes. The only network call notebook
  ever makes is fetching the chapter markdown.

### Graphic stories (P5)

- Live at `/graphic-stories/{slug}` with a custom layout: panel-by-panel
  reader with keyboard nav and swipe-to-advance on mobile.
- Each story credits Studio Whence prominently.
- A story adapted from a corpus work links back to the source chapter,
  and that chapter's frontmatter has a `graphic_adaptations:` field
  surfacing the link.
- Studio Whence permission gate: a `LICENSE.md` per story explicitly
  marking it public-good. Stories without that file fail the build.

### Translation attribution (P5b)

- Every chapter page has a "Translated by Thothica" badge in the chapter
  header, linking to thothica.com.
- The `translator:` frontmatter field is the source of truth. Values:
  `thothica` (current), `native` (future English originals), or any
  free-form string for third-party translators (with optional URL via
  `translator_url:`).
- The site's About page describes the Thothica + Studio Whence + Falsafa
  relationship explicitly.
