## Per-chapter imagery decision-point

**What:** Decide whether `/works/{slug}/{chapter}` pages need AI-generated art or
whether the typographic chapter header (drop cap + chapter number + title in
display serif) carries the page.

**Why:** V1 ships covers only (38 images). The original plan included ~1,673
chapter images. Codex flagged this as strategically off — chapter art adds cost,
QA burden, repo bloat, and aesthetic risk to a text product where the content
itself is the draw. The eng review deferred the decision rather than killing
the option entirely.

**Pros of doing chapter imagery (later):** Visual richness, distinguishes
Falsafa from text-only digital editions, ties into Studio Whence's house
aesthetic for a unified cultural-project feel.

**Cons of doing chapter imagery:** ~$15 + 25 min compute + ~3-5 hours QA pass
to weed out misses, repo bloat to ~150-200MB (move to Cloudflare R2 if it
crosses 500MB), prompt drift over time, ~5-10% aesthetic miss rate.

**Trigger:** 30 days after V1 launch. Read the analytics, read the user
feedback, sit with your own reading experience. If chapter pages feel flat,
build the pipeline. If typographic headers carry the page, save the QA burden.

**Where to start:** Re-read this design doc's "Phase 4: AI imagery" section
for the original spec. Adapt to chapter-only scope. Pipeline is identical to
the cover-image script with a different prompt template.

**Depends on:** V1 ships and accumulates ~30 days of real reading experience.

## Side-by-side bilingual reader (when paired content exists)

**What:** Implement the parallel-text reading mode that the schema is already
forward-designed for via the `paired_chapter:` frontmatter field.

**Why:** A premium reading platform for translated works has a credibility
ceiling without parallel-text rendering. NYRB digital editions, Loeb Classical
Library, Norton Critical Editions — all serious editions of translated works
let readers see the original. Today's `works.json` ships single-language
chapters; readers can't see Iqbal's Urdu next to Thothica's English rendering.

**Pros:** Closes the credibility gap, especially for serious classics readers.
Distinguishing feature against any other AI-translation reading platform.

**Cons:** Requires Thothica content workstream to produce paired translations
of the original-language works (Andreas English, Sanskrit Āṅgirasa English)
and originals for the translated-only works (Iqbal Urdu source).

**Trigger:** Thothica's content workstream produces paired translations for
>= 5 works. (Below 5, the side-by-side mode is too patchy to feel intentional.)

**Where to start:** Schema is ready — every chapter file already has the
optional `paired_chapter:` field. Convert script just needs to populate it
when paired data exists. Reader component needs a parallel-text layout and
a toggle. Astro view-transitions handle the toggle smoothly.

**Depends on:** Thothica content production. Independent of any V1 code.

## Vendor deep-link drift monthly cron

**What:** A scheduled GitHub Action that opens `https://claude.ai/new?q=<test>`
and `https://chat.openai.com/?q=<test>` in a headless browser and asserts the
pre-fill behavior still works. Posts to a Slack channel / opens an issue if
either URL format changes.

**Why:** The notebook export feature uses these vendor URLs. They are not
contractual APIs. They can change without notice. The eng-review failure-mode
audit flagged this as a silent-breakage risk: button looks fine, vendor changes
URL format, button silently does nothing useful. Without monitoring, you find
out from a user report.

**Pros:** Catches regression in <24 hours instead of "whenever someone tells
you." Cheap to operate (~$0.01/month in CI minutes).

**Cons:** Headless browser tests are flaky. False positives are noise. Have
to tune the assertion to be lenient enough to ignore minor vendor UI churn
but strict enough to catch real format breaks.

**Trigger:** Set up in V1 launch week, run from launch day forward.

**Where to start:** GitHub Actions workflow with `playwright` action,
runs on cron `0 9 1 * *` (1st of each month, 9am UTC). Asserts encoded URL
loads to a chat with the test prompt visible in the input. On failure, opens
a GitHub issue with screenshots.

**Depends on:** V1 ships with the deep-link export feature.

## Stanza-segmentation pass for poetry chapters (V1 BLOCKER)

**What:** Re-format the `chapter.content` field for all poetry-flavored chapters
(Iqbal Bang-E-Dara × 300, Ghalib Diwan × 717, Zauq Diwan × 180, plus Indic smṛti
texts) so each stanza is an explicit structural unit the verse-layout renderer
can iterate over.

**Why:** /plan-design-review surfaced that the current JSON has poems as flat text
with implicit line breaks. The locked verse-layout renderer (variant C) shows
stanzas as discrete blocks with breathing room between them. Without explicit
stanza segmentation, the verse layout renders 1,400 chapters as flat blocks,
defeating the design.

**Pros:** The verse renderer actually delivers the intended reading experience.
Search and citation can target individual stanzas. MCP `get_passage` becomes
stanza-precise.

**Cons:** ~1,400 chapters need a content-cleanup pass. Some can be auto-segmented
from line-break heuristics; others need manual review.

**Approach:** Three-pass plan. (1) Heuristic auto-segmentation in `convert.ts`:
double-newline = stanza boundary, single-newline = line break. Run against all
poetry chapters, write to `corpus/works/.../chapters/*.md` with explicit stanza
markers. (2) Quick visual review pass in the dev environment — eyeball 50 random
chapters, flag misformatted ones. (3) Manual cleanup of flagged chapters.

**Trigger:** Phase 1 (Convert + Catalog). This is V1 blocker, not deferred.

**Where to start:** Add stanza-detection logic to `scripts/convert.ts`. Mark
`layout: verse` chapters with explicit `## Stanza N` markers in the markdown body
or a `stanzas:` array in frontmatter (decide based on what's easier to render in
Astro).

**Depends on:** Schema decision on stanza representation (markdown convention
vs frontmatter array).

## Click-prompt content for home page (V1 BLOCKER if home ships with this section)

**What:** Write 8-12 click-prompts that appear in the home page's "Make your own
path" section. Each prompt deep-links to Claude or ChatGPT with the falsafa MCP
attached, asking the LLM to generate a curated reading path live.

**Why:** /plan-design-review locked the home page IA as Featured + Curated paths
+ Click-prompts + Era timeline. The click-prompts section is decoration if
empty, function if filled. This is the moment that takes "premium reading
platform" past static, into AI-leveraged exploration.

**Pros:** Unique to Falsafa. Showcases the MCP. Lowers the bar for readers who
don't know where to start. Each prompt is a tiny piece of editorial taste that
travels with the user into their LLM session.

**Cons:** Each prompt needs to be sharp enough that the LLM produces a good
reading path. Bad prompts → bad output → user bounces.

**Examples to start from:**
- "I'm new to philosophy. Where should I begin?"
- "Show me how different traditions treat the concept of dharma / duty."
- "I want to read about justice across cultures."
- "What's the connection between Old English Christianity and Sanskrit dharma?"
- "I have one hour. Give me one chapter that will stay with me."
- "Compare how Iqbal and Ghalib treat the divine."
- "Where does the corpus discuss death?"
- "Recommend a reading path for someone who loved Rumi."
- "What's the best entry point to medieval European literature?"
- "Show me the political-theory works in chronological order with one paragraph each."

**Trigger:** Phase 3 (Astro reading site).

**Where to start:** Draft prompts in `corpus/home/click-prompts.yaml`. Each entry
has `prompt: <string>`, `label: <button text>`, `category: <philosophy | history
| literature | dharma | etc>`. Site reads the YAML at build time and renders the
section.

**Depends on:** Phase 2 (MCP server) ships first, otherwise the prompts have
nothing to call.

**What:** Write 3-4 hand-curated reading paths for the home page's "Where to
start" section. Each path is a sequence of 3-5 chapters from the corpus with a

**Why:** /plan-design-review locked the home IA. This section is the editorial

**Trigger:** Phase 3 (Astro reading site).

**Where to start:** Write paths in `corpus/home/curated-paths.yaml`. Each path

chapter_n, why}]`. Site reads at build time and renders.

**Depends on:** Phase 1 (Markdown corpus exists, slugs are stable).

## Cover imagery prompt template per era (V1 BLOCKER for Phase 4)

**What:** Draft per-era prompt templates that drive the Replicate Flux dev
generation for the 38 cover images. Each template defines the era's visual
identity in concrete enough terms that Replicate produces coherent results.

**Why:** /plan-eng-review locked Phase 4 as covers-only with hash-cached
generation. /plan-design-review flagged that "per-era painterly style" without a
written template produces generic results. The brief has to be sharp.

**Pros:** Coherent visual identity across the catalog. Era-appropriate covers
that actually evoke the era, not generic fantasy art.

**Cons:** Prompt-engineering takes iteration. Some eras (e.g., Sanskrit smṛti
literature) don't have an obvious visual canon.

**Approach:** One template file per era at `scripts/image-prompts/{era}.json`.
Each template has: `style_guide` (paragraph describing the visual mood),
`reference_artists` (Studio Whence aesthetic anchors), `palette` (hex colors),
`composition_rules`, `negative_prompt` (what to avoid). The script substitutes
work-specific tokens (title, author, language) into the template.

**Eras to template:** Antiquity / Medieval / Early Modern / Modern / Indic /
Anglo-Saxon (the actual era set in the data).

**Trigger:** Phase 4 (AI imagery).

**Where to start:** Run a small experiment. Generate 3 covers for one work in
each era using a draft template. Iterate on the template until covers feel
era-appropriate. Document the working templates.

**Depends on:** Phase 1 (works have era frontmatter).

## Held chapter splits — per-variant parser dispatch

**What:** 6 works need their chapter splits done but each variant uses a
different marker syntax — the current orchestrator binds one parser per
work, which can't handle this. We need per-variant parser dispatch.

**Affected works:**

| Work | Translation pattern | Transliteration / original pattern |
|------|--------------------|-----------------------------------|
| Bṛhaspati Smṛti | `## Chapter N:` markdown | `// Brh_chapter,section.verse //` (3-part) |
| Kātyāyana Smṛti | `## ` markdown (80 sections) | `// K_NNN //` (sequential, no chapter index) |
| Kunjarakarna Dharmakathana | `## Canto N` markdown | numbered verses (`1.`, `2.`) under `Canto N.` lines |
| Vīramitrodaya | `### [...]` markdown | `{MV-S_N}` curly tokens + `// N //` numeric verses |
| Aṅgirasa Smṛti | phantom translator-added markers (`Ang_11.x` cross-refs) | clean `// Ang_1.N //` |
| Nāradasmṛti | `(Nar. M1.1)` parenthesized format | clean `// Nar_1.N //` |

**Why deferred:** the existing splitter (`scripts/chapter-splitting/lib/orchestrator.ts`)
takes one `parseFunc` per work. To handle these we need either:
(a) per-variant parseFunc dispatch (parser keyed on `(work_slug, variant_filename)`),
or (b) a more general marker-detection system that tries multiple parsers and picks
the one with the best variant agreement.

**Impact:** until fixed, these 6 works ship as single-chapter (Cynewulf-style
title-page redirect). Reading is fine; URL anchors for annotations will need to
move once the splits land. **This is a real annotation-blocker for these 6 works.**
Annotations on the other ~32 works are unaffected.

**Where to start:** extend `ParseFunc` in `lib/orchestrator.ts` to accept a
`filename` (already passes it) and dispatch via a per-variant config map in the
TYPE_C entry. The variant-level parsers needed:
- 3-part TYPE_A regex variant (`Brh_C,S.V`) — extension of `lib/parser.ts`
- Sequential-no-chapter regex (`K_NNN`) — synthetic chapter from sequence
- Bracketed-section parser (`[saṃvatsara…]`) — like markdown headings but with `[` delim
- Inline citation filter for Aṅgirasa (translator's `Ang_11.x` cross-refs)
- Parenthesized format parser (`(Nar. M1.1)`) for Nāradasmṛti translation

**Trigger:** before annotations ship. URL stability matters most when there's a
notebook full of highlighted passages anchored to specific chapter slugs.

**Depends on:** orchestrator API change in `lib/orchestrator.ts` (small).

## Paragraph-level Pagefind upgrade

**What:** Re-index Pagefind at paragraph granularity instead of chapter granularity,
so a search for "courage" returns the *paragraph* that contains it, not just the
chapter. Hits scroll-spy to a fragment anchor on click.

**Why:** Chapter-level indexing is the V1 default — fewer docs, smaller index
(~1MB vs ~5–10MB at paragraph granularity), simpler URL routing (no fragment
anchors). It works, but it forces the user to skim a chapter for the passage
that matched. For the long Comte volumes (28 chapters × ~5,000 words), this is a
real friction.

**Trigger:** When user feedback or session analytics show readers giving up on
search results that point to a chapter without a fragment anchor. Or when annotated
chapters become large enough that `Cmd-F` after the click feels broken.

**Where to start:** Pagefind has first-class subdivision support. Mark the
`<article>` body with `data-pagefind-body` and individual paragraphs with
`data-pagefind-meta`. Add fragment anchors in `ChapterBody.astro` so clicks on
result snippets land at the right paragraph.

**Depends on:** V1 ships with chapter-level Pagefind first.

## Embeddings-based cross-link upgrade

**What:** Replace the build-time TF-IDF cross-link index with embeddings-based
similarity (sentence-transformer or OpenAI ada-002 over chapter summaries),
producing semantically richer "see also" links.

**Why:** TF-IDF over ~750 docs is enough for V1 — it catches keyword overlap and
delivers reasonable cross-tradition matches. But it misses paraphrases (e.g.,
"the divine" in Iqbal vs. "godhead" in Cynewulf). Embeddings would catch these.

**Pros:** Better cross-tradition find_related results. The MCP tool becomes
substantially more interesting for comparative questions ("what does each
tradition say about X?").

**Cons:** Build-time dependency on an embedding model (either bundled MiniLM
~80MB binary, or a paid API call per chapter). Embeddings index re-generates
on every corpus change. Quality is hard to evaluate — bad embeddings produce
worse results than no cross-links.

**Trigger:** Corpus crosses ~5K chapters, OR find_related becomes a primary
discovery surface that justifies the model dependency, OR a user complains
about cross-tradition discovery quality.

**Where to start:** Bun + @huggingface/transformers running MiniLM-L6-v2 locally.
Compute one embedding per chapter (~768 dims), pairwise cosine, write the same
`corpus/cross-links.json` shape so MCP doesn't need to change.

**Depends on:** V1 ships with TF-IDF cross-link first (so we have a baseline to
compare against).

## RSS feed for new chapters / works

**What:** Build-time RSS generation, exposed at `/feed.xml`, listing recent
chapters and new works added to the corpus.

**Why:** Classics and philosophy readers are RSS-native. A Substack-flavored
update channel (without paying Substack) is the right shape. Audience for
RSS overlaps strongly with the audience for serious humanities reading.

**Pros:** Cheap, durable, no algorithmic platform involved. Builds an
audience that returns.

**Cons:** Updates need to be frequent enough to justify subscribing. If
the corpus is stable for months, the RSS feed is stale and unsubscribed.
Tied to a content cadence.

**Trigger:** Post-V1 launch when there's a content cadence in place.

**Where to start:** Astro has first-class RSS support via `@astrojs/rss`.
Generate from `corpus/manifest.json` recent additions. Linked from the
site footer. Submit to relevant aggregators.

**Depends on:** V1 launches with at least the catalog. RSS is a follow-up
weekend, not a launch blocker.
